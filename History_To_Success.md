# Apple II 磁碟啟動成功之路：技術對話精華錄 (2026-03-12)

本筆記記錄了從「丟失代碼」到「精確位元級模擬」的還原歷程，解決了磁碟讀取位移與時序同步的核心問題。

## 1. 核心突破：從位元組到「位元」 (Bit-Level Emulation)
*   **問題**：原有的 32 週期位元組模擬過於粗糙，導致 CPU 的 7 週期輪詢迴圈（`LDA $C08C,X / BPL`）經常錯過資料窗口，造成讀取亂碼或全 0。
*   **解決方案**：在 `disk2.rs` 中實作 **P6 Sequencer 移位暫存器**。
    *   **時序**：每 4 個 CPU 週期位移 1 個位元（1 MHz = 1μs/cycle, Disk II = 4μs/bit）。
    *   **邏輯**：當移位暫存器（Shift Register）的 MSB (Bit 7) 變為 1 時，觸發 `latch_ready` 訊號。
    *   **結果**：成功自動對準同步位元（Sync Bytes, $FF），不再需要手動計算物理偏移。

## 2. 物理編碼：LinApple 工業標準
*   **映射順序**：採用 `bits 5,4 <- data[i]`, `bits 3,2 <- data[i+86]`, `bits 1,0 <- data[i+172]` 的 LinApple 標準。
*   **物理反轉**：確認 P5 ROM 的 `DEY` 循環特性，Secondary Buffer (前 86 nibbles) 必須以 `85..0` 的物理順序寫入磁軌。
*   **發現規律**：在位元級模擬下，觀察到解碼後的資料呈現穩定的 `-1` 偏移（例如 `00` 代替 `01`, `A4` 代替 `A5`）。這證明了索引映射已正確，僅剩最後 1-bit 的硬體同步微調。

## 3. ROM 完整性驗證
*   **警訊**：發現部分下載的 `.bin` 文件實為 HTML 網頁（`<!DOCTYPE html>`）。
*   **確認**：經 `hex dump` 驗證，`APPLE2PLUS.ROM` 與 `DISK2.ROM` 內容包含正確的 BASIC 錯誤訊息及 Slot 6 啟動代碼，排除了底層資料錯誤的可能。

## 4. CPU 時序修正
*   **Page Cross Penalty**：針對 `AbsoluteX`, `AbsoluteY`, `IndirectY` 等定址模式，實作了跨頁邊界增加 1 週期的懲罰，確保 RWTS 時序精確。
*   **PHP/PLP**：修正了狀態暫存器在堆疊壓入/彈出時的 Bit 4/5 處理，符合 6502 真實硬體行為。
