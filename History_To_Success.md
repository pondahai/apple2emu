# Apple II 磁碟啟動成功之路：技術對話精華錄 (2026-03-12)

本筆記記錄了從「丟失代碼」到「精確位元級模擬」的還原歷程，解決了磁碟讀取位移與時序同步的核心問題。

## 1. 核心突破：位元級移位暫存器 (Bit-Level Shift Register)
*   **物理模擬**：捨棄了粗略的 32 週期位元組模型，改為每 4 個 CPU 週期移動 1 個位元的精確模擬。
*   **同步機制**：實作了 P6 Sequencer 的位位元移邏輯。當移位暫存器的 Bit 7 變為 1 時，觸發資料就緒（Ready）。這解決了 CPU 輪詢時經常錯過資料窗口的問題。
*   **對準結果**：系統已能穩定自動對準磁區標頭（Prologue: D5 AA 96/AD），並正確跳轉至磁碟執行段（$0801）。

## 2. 物理編碼：黃金對齊公式 (Golden Formula)
*   **映射分組**：`bits 1,0 <- swap(data[i])`, `bits 3,2 <- swap(data[i+86])`, `bits 5,4 <- swap(data[i+172])`。
*   **位元互換 (Internal Swap)**：由於 6502 `LSR/ROL` 解碼特性，編碼時必須執行 `(b&1)<<1 | (b>>1)&1`。
*   **物理順序**：次要緩衝區（前 86 nibbles）以 `85..0` 反序寫入；主要緩衝區（後 256 nibbles）以正序寫入。
*   **XOR 鏈累積**：`encoded = raw ^ last; last = raw;`（使用原始 6-bit 值作為種子）。
*   **磁碟控制器**：實作「毀滅性讀取」，即讀取 `$C0EC` 後立即清空 Bit 7，確保 RWTS 輪詢窗口的唯一性。

**目前狀態**：模擬器已完全找回並優化了昨日的成功邏輯。目前的引導序列能穩定通過校驗並精確對齊 `$0800` 資料段。


## 3. 已修復的關鍵細節
*   **CPU 時序**：實作 Page Cross Penalty（跨頁增加 1 週期），修正 PHP/PLP 旗標處理。
*   **ROM 驗證**：排除了 HTML 假 ROM 的干擾，確認 `APPLE2PLUS.ROM` 與 `DISK2.ROM` 資料真實性。
*   **穩定性**：目前的架構能穩定通過 Boot ROM 的 Checksum 校驗，不再陷入死循環。
